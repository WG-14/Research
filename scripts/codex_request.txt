Required design changes:

1. Live dry-run broker truth guard

Implement a guard that applies before any balance split mismatch, broker-local convergence, recovery blocker, or resume safety decision.

For `MODE=live`, `LIVE_DRY_RUN=true`, `LIVE_REAL_ORDER_ARMED=false`:

* Expected broker truth source: `accounts_v1_rest_snapshot`.
* Expected simulation balance source: `dry_run_static`.
* If `/v1/accounts` succeeded, use accounts-v1 cash/asset evidence for broker truth.
* If a candidate broker truth snapshot has `source_id=dry_run_static`, do not use it for broker-local convergence.
* Emit structured evidence:

  * `balance_authority=accounts_v1_rest_snapshot`
  * `broker_truth_source=accounts_v1_rest_snapshot`
  * `simulation_balance_source=dry_run_static`
  * `accounts_v1_cash=<actual KRW>`
  * `dry_run_static_cash=<START_CASH_KRW>`
* If an unsafe source is about to be used, emit:

  * `LIVE_DRY_RUN_BROKER_TRUTH_SOURCE_VIOLATION`
  * `expected=accounts_v1_rest_snapshot`
  * `got=dry_run_static`
  * `simulation_balance_source=dry_run_static`
* Decide whether this should be a hard failure or a fail-closed warning depending on the command path:

  * For resume/recovery safety: fail closed.
  * For diagnose/reporting: warn loudly and render both sources.

2. Recovery/reconcile integration correctness

Patch the actual recovery/reconcile readiness path, not only helper functions. The EC2 failure mode must be impossible:

```text

broker=1000000
reason_code=PORTFOLIO_BROKER_CASH_MISMATCH
```

when accounts-v1 returned KRW `395113.99879`.

3. Broker-diagnose output redesign

Update `broker-diagnose` so live dry-run output cannot mislead an operator.

In live dry-run, output should contain separate sections or explicit fields:

```text

accounts_v1_cash=395113.99879
accounts_v1_asset_qty=0.00000000

```

Do not render `dry_run_static` as broker balance truth. If it is shown, label it as simulation only.

4. Trusted order-rule snapshot audit

Audit every call to `fetch_latest_order_rule_snapshot()`.

* If the usage is for canary baseline, live fallback authority, live preflight, broker readiness, or runtime safety, replace it with `fetch_latest_trusted_order_rule_snapshot()`.
* If the usage is only for audit/history display, keep it but name/comment it as audit-only.
* Add regression coverage proving `fallback_used=1` / `AUTH_JWT_VERIFICATION` snapshots are ignored as authority/baseline.

5. Fallback snapshot quarantine metadata

When persisting fallback snapshots caused by auth/transport failure, ensure `source_json` contains clear untrusted metadata:

```json

  "trust_level": "auth_failed_quarantine",
  "eligible_for_canary_baseline": false,
  "eligible_for_live_order_authority": false,
  "operator_action": "restore_private_api_or_review_fallback_before_live"
}
```

For non-auth local fallback, use an equally explicit untrusted level such as `local_fallback_untrusted`.

6. Env contract/lint hardening

Add a command or extend `broker-diagnose` to run an env contract lint. It must be safe for logs. It may output lengths and safe hash prefixes, never secret values.

Flag at least:

* placeholder values like `<verified-live_dry_run-profile>`

* deprecated ignored keys such as `BUY_PRICE_NONE_MARKET_TO_PRICE_ALIAS_ENABLED`

* risky live limits such as `MAX_DAILY_ORDER_COUNT=500`
