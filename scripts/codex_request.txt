## Hypotheses to verify or falsify

Before coding, inspect the repository and classify each hypothesis as confirmed, partially confirmed, or rejected.

### Hypothesis 1: `scenario_policy` exists but is not actually enforced at the candidate level

Expected current risk:

* `execution_model.scenario_policy` can contain values such as `must_pass_base_and_survive_stress`.
* `validation_protocol.py` may evaluate each execution scenario independently and append one candidate row per scenario.
* `_report_payload` may choose the first `acceptance_gate_result == PASS` row as best candidate.
* This may allow a parameter candidate to pass under one scenario while failing under another required scenario.

Concrete failure mode to test:

```text
candidate_A + base scenario = PASS
candidate_A + stress scenario = FAIL
candidate_B + stress scenario = PASS
```

The final result must not treat `candidate_A` as promotion-eligible, and must not let a stress-only or base-only pass satisfy `must_pass_base_and_survive_stress`.

### Hypothesis 2: stochastic `StressExecutionModel` may be candidate-order dependent

Expected current risk:

* `StressExecutionModel` owns mutable `random.Random(seed)` state.
* If the same model instance is reused across candidates or splits, earlier candidates consume RNG events and change later candidates’ fills.
* Candidate results can depend on parameter-space enumeration order rather than candidate identity, scenario, split, and seed.

Concrete failure mode to test:

* Run the same manifest twice with the same seed.
* Reorder candidate enumeration or parameter-space ordering.
* The same `parameter_candidate_id` should produce the same stress execution metadata.
* If results differ only because another candidate was evaluated earlier, the implementation is unsafe.

### Hypothesis 3: execution calibration may not be bound to the manifest’s market and interval

Expected current risk:

* Calibration artifacts include `market` and `interval`.
* `compare_calibration_to_scenario` may only compare p90/p95 slippage, latency, and model breach.
* A calibration artifact from a different market or interval could be accidentally accepted.

Concrete failure mode to test:

* Research manifest uses `KRW-BTC` / `1m`.
* Calibration artifact says `KRW-ETH` / `5m`.
* The gate must fail closed with explicit mismatch reasons.

### Hypothesis 4: calibration content hash may be optional even when calibration is required

Expected current risk:

* `validate_calibration_artifact` may verify `content_hash` if present but allow missing hash.
* For `calibration_required=true` or promotion evidence, hashless calibration should not be acceptable.

Concrete failure mode to test:

* Provide otherwise valid calibration artifact without `content_hash`.
* When calibration is required, the candidate must fail with `execution_calibration_content_hash_missing`.

### Hypothesis 5: final holdout may not be required for promotion

Expected current risk:

* `dataset.final_holdout` may be optional in manifests.
* Promotion may validate backtest and walk-forward evidence but not require final-holdout evidence.
* For a promotion artifact that can lead toward paper/live profile consideration, missing final holdout should be explicit and fail-closed unless the repository already has a more precise equivalent policy.

Concrete failure mode to test:

* Candidate passes validation and walk-forward, but manifest/report lacks final-holdout evidence.
* Promotion must be refused or must require an explicit manifest policy that forbids live/promotion use without final holdout.

## Required patch priorities

### P0: Enforce scenario policy at the aggregated candidate level

Refactor research result construction so scenario results are grouped by stable parameter candidate identity.

Required behavior:

* A parameter candidate should have one top-level candidate record.
* That record should contain `scenario_results[]`.
* Each scenario result should include:

  * scenario identity;
  * execution model payload;
  * model params hash;
  * train metrics;
  * validation metrics;
  * final holdout metrics if present;
  * walk-forward metrics if applicable;
  * regime gate result;
  * execution calibration gate;
  * scenario acceptance result;
  * scenario fail reasons.
* The top-level candidate should compute final `acceptance_gate_result` from `execution_model.scenario_policy`.

For `scenario_policy == "must_pass_base_and_survive_stress"`:

* The same parameter candidate must pass all required scenarios.
* If any scenario fails, top-level candidate must fail.
* Top-level `gate_fail_reasons` must include scenario-level failures in a machine-readable form.
* The selected `best_candidate_id` must be an aggregated candidate id, not a scenario row id.
* Promotion must verify the aggregated candidate evidence, not only one scenario result.

If the repository supports other scenario policies, preserve them. If not, implement only:

```text
legacy_cost_model_single_pass
single_scenario
must_pass_base_and_survive_stress
```

Fail clearly on unsupported scenario policies.

Suggested fail reasons:

```text
scenario_policy_unsupported
scenario_policy_required_scenario_failed
scenario_policy_no_passing_base_scenario
scenario_policy_no_passing_stress_scenario
scenario_result_missing
```

### P0: Make stochastic stress execution deterministic and candidate-order independent

Patch `StressExecutionModel` or the validation runner so stochastic behavior is reproducible by candidate/split/scenario identity.

Acceptable designs:

1. Create a fresh execution model instance for every candidate/split/scenario evaluation and derive the seed from:

   * manifest hash;
   * scenario params hash;
   * parameter candidate id;
   * split name;
   * base seed.

2. Or make `StressExecutionModel.simulate()` derive randomness from immutable per-request material, including:

   * model seed;
   * scenario hash;
   * candidate id;
   * split name;
   * trade index or signal timestamp;
   * side.

Do not rely on shared mutable RNG state across candidates.

Required behavior:

* Same manifest + same seed + same candidate + same split must produce identical trade-level execution metadata.
* Reordering parameter candidates must not alter a candidate’s stress fills.
* Different seeds may produce different stress outcomes, but the seed and model params hash must make the result auditable.

Add observability fields if needed:

```text
stress_seed_material
stress_seed_hash
scenario_id
split_name
```

### P1: Bind execution calibration artifacts to manifest market, interval, and content hash

Patch calibration comparison and validation.

Required behavior:

* `compare_calibration_to_scenario` should accept expected market and interval, or equivalent manifest context.
* If calibration market differs from manifest market, fail with:

```text
execution_calibration_market_mismatch
```

* If calibration interval differs from manifest interval, fail with:

```text
execution_calibration_interval_mismatch
```

* If `calibration_required=true`, missing `content_hash` must fail with:

```text
execution_calibration_content_hash_missing
```

* If `content_hash` is present but invalid, preserve the existing mismatch failure behavior.
* If calibration is optional and strictness is `warn`, missing hash may be warning-only, but the report must make this explicit.
* Promotion must not accept required calibration evidence without a valid content hash.

### P1: Require final-holdout evidence for promotion

Patch promotion gating so final holdout evidence is not silently optional for promotion.

Acceptable designs:

1. Add `final_holdout_required_for_promotion: bool = true` to `AcceptanceGate`, defaulting to true for new manifests.
2. Or enforce final holdout unconditionally in `promote_candidate`.
3. Or implement a clearly named manifest policy, but do not default to unsafe promotion.

Required behavior:

* Candidate payload must include whether final holdout was present.
* Promotion must fail when required final-holdout evidence is missing.
* Fail reason should be explicit:

```text
final_holdout_evidence_missing
```

* If final holdout exists, promotion should verify its metrics are present and profile hash covers the evidence or its relevant hash.
