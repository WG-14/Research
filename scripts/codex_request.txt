### Hypothesis A — `promote_profile_mode()` may not re-check parent source/evidence drift before creating a child profile

`verify_profile_source_promotion()` appears to verify both:

* source promotion artifact integrity
* evidence artifact integrity

However, the caller path may still be:

```python
parent = load_approved_profile(profile_path)
child = promote_profile_mode(...)
```

That is safe only if `promote_profile_mode()` itself calls one of the following at the beginning:

```python
verify_profile_source_promotion(parent_profile)
```

or the equivalent explicit pair:

```python
verify_profile_source_artifact(parent_profile)
verify_profile_evidence_artifacts(parent_profile)
```

Required action:

* Open `src/bithumb_bot/approved_profile.py`.
* Inspect the full `promote_profile_mode()` body.
* If parent source/evidence verification is already performed before child construction, do not duplicate it.
* If it is missing or occurs after child construction, patch it so the parent profile is verified before any child profile is created.

Required behavior:

* `profile-promote` from `paper` to `live_dry_run` must fail if the parent paper profile’s source promotion artifact has drifted.
* `profile-promote` from `live_dry_run` to `small_live` must fail if the parent live-dry-run profile’s source promotion artifact has drifted.
* `profile-promote` from `live_dry_run` to `small_live` must fail if the parent profile’s paper validation evidence has drifted.
* If live readiness evidence is already attached to a parent in any supported transition, its drift must also be detected.
* Failure reason must be specific, such as:

  * `source_promotion_content_hash_mismatch`
  * `paper_validation_evidence_content_hash_mismatch`
  * `live_readiness_evidence_content_hash_mismatch`

Priority: **high**.
