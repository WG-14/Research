You are Codex working directly inside the `WG-14/bithumb-bot` repository on the `main` branch.

Your task is to perform the next stabilization patch after the prior research-validation patches. The previous P0/P1 structural issues appear mostly resolved: scenario aggregation exists, `scenario_role` is now first-class, single-scenario policy defaulting was improved, calibration warnings are included in candidate/promotion artifacts, and pytest reportedly passes. Do not treat this as finished. The remaining work is to make operator-visible warnings harder to miss, reject impossible policy/role combinations earlier, and add full-pipeline regression coverage.

This patch is not a large redesign. It is an operational observability and policy-prevalidation hardening patch.

## Mission

Lock down the research-promotion pipeline so that:

1. an operator running the promotion CLI cannot miss warning-mode calibration breaches;
2. impossible `scenario_policy` + scalar `scenario_role` combinations fail at manifest-parse time instead of after a long research run;
3. the actual `run_research_backtest()` path is covered by integration tests, not only unit tests of policy helpers;
4. docs clearly explain scalar `scenario_role` behavior and warning-mode calibration semantics.

Do not weaken any existing fail-closed behavior.

## Scope boundaries

Do not rewrite the research pipeline.

Do not change live-trading execution behavior.

Do not relax promotion gates, final-holdout requirements, scenario-policy enforcement, calibration-required enforcement, profile-hash verification, walk-forward checks, or regime-policy checks.

Do not remove legacy `cost_model` compatibility.

Do not fake exchange data, orderbook data, bid/ask spread, or fill data.

Do not treat warning-mode calibration failures as hard failures unless the existing policy already requires that. The goal is to make warnings visible, not to change optional-warn semantics into fail-closed semantics.

Do not claim success based only on unit tests. Add or verify full integration coverage.

## Hypotheses to verify or falsify

Before patching, inspect the repository and classify each hypothesis as confirmed, partially confirmed, or rejected.

### Hypothesis 1: promotion CLI still hides warning-mode calibration breaches

Expected current risk:

* `promote_candidate()` writes top-level artifact fields such as:

  * `has_execution_calibration_warning`;
  * `execution_calibration_warning_reasons`;
  * `promotion_warnings`.
* But `cmd_research_promote_candidate()` may still print only:

  * `gate_result`;
  * `artifact_path`;
  * `content_hash`;
  * `operator_next_step`.

If true, an operator who only reads CLI output can miss an optional calibration breach even though the artifact contains it.

Required behavior:

* The promotion CLI must print warning information directly when promotion succeeds.
* At minimum, print:

  * `has_execution_calibration_warning=1/0`;
  * `execution_calibration_warning_reasons=...`;
  * `promotion_warnings=...`.

### Hypothesis 2: impossible scalar `scenario_role` + `must_pass_base_and_survive_stress` combinations are only rejected after running research

Expected current risk:

* If an execution model generates multiple scenarios and the manifest supplies scalar `scenario_role="base"`, all scenarios become base.
* If `scenario_policy="must_pass_base_and_survive_stress"`, the candidate will eventually fail because there is no stress scenario.
* This is fail-closed but late. It wastes research runtime and makes the manifest contract less clear.

Required behavior:

* If the manifest explicitly sets `scenario_policy="must_pass_base_and_survive_stress"` and explicitly sets scalar `scenario_role` so all generated scenarios have only one role, reject at manifest parse time.
* Error message should be explicit:

```text
execution_model.scenario_role conflicts with must_pass_base_and_survive_stress
```

Do not reject this combination when the policy is `single_scenario`.

Do not reject derived roles where generated scenarios include both base and stress.

Do not reject legacy `cost_model` manifests.

### Hypothesis 3: full `run_research_backtest()` scenario aggregation may still lack integration coverage

Expected current risk:

* Unit tests cover `_apply_scenario_policy()` and manifest parsing.
* But there may not be a full integration test that exercises:

  * SQLite candle fixture;
  * manifest parsing;
  * scenario generation;
  * `run_research_backtest()`;
  * candidate aggregation;
  * scenario result reporting;
  * candidate profile hash generation;
  * promotion refusal for failed scenario policy.

Required behavior:

* Add or verify a full-pipeline test using synthetic SQLite candle data.
* It should call `run_research_backtest()` directly.
* It should assert the final report structure, not only helper functions.
* It should attempt `promote_candidate()` and verify refusal when scenario-policy evidence fails.

### Hypothesis 4: docs may not clearly explain scalar `scenario_role` behavior and warning calibration visibility

Expected current risk:

* Docs may mention scenario roles and calibration warnings generally.
* But they may not clearly state that scalar `execution_model.scenario_role` applies to every generated scenario.
* They may not explain that warning-mode calibration breaches are allowed but must be operator-visible in promotion artifacts and CLI output.

Required behavior:

* Update docs/runbooks with repository-specific guidance.
* Do not write generic trading advice.

## Required patches

### P1: Print promotion warnings in the CLI

Patch `src/bithumb_bot/research/cli.py`, specifically `cmd_research_promote_candidate()`.

After successful promotion, print:

```text
has_execution_calibration_warning=1/0
execution_calibration_warning_reasons=<comma-separated list or none>
promotion_warnings=<comma-separated list or none>
```

Use the artifact fields produced by `promote_candidate()`.

Suggested format:

```text
  has_execution_calibration_warning=1
  execution_calibration_warning_reasons=execution_calibration_p95_slippage_exceeds_assumption
  promotion_warnings=execution_calibration_p95_slippage_exceeds_assumption
```

If no warnings exist:

```text
  has_execution_calibration_warning=0
  execution_calibration_warning_reasons=none
  promotion_warnings=none
```

Tests required:

* Promotion CLI output includes warning fields when warn-mode calibration breach exists.
* Promotion CLI output includes `none`/`0` values when no warnings exist.
* Required calibration failure still refuses promotion and does not print a successful promotion block.

### P1/P2: Reject impossible scalar role/policy combinations at manifest parse time

Patch `src/bithumb_bot/research/experiment_manifest.py`.

Required behavior:

* After generated scenarios are built and the final scenario policy is known, validate policy/role consistency.
* If:

  * `scenario_policy == "must_pass_base_and_survive_stress"`;
  * `scenario_role_source == "manifest"` for all generated scenarios;
  * all generated scenarios have only one role, either all `base` or all `stress`;
  * scenario count is greater than one;
* then raise `ManifestValidationError` with:

```text
execution_model.scenario_role conflicts with must_pass_base_and_survive_stress
```

Do not block:

* `single_scenario` policy with scalar role;
* legacy `cost_model` policy;
* derived roles that include both base and stress;
* explicit configurations that can actually produce both required roles.

Tests required:

* Multi-scenario manifest with `scenario_policy="must_pass_base_and_survive_stress"` and scalar `scenario_role="base"` is rejected.
* Same with scalar `scenario_role="stress"` is rejected.
* Multi-scenario manifest with `scenario_policy="single_scenario"` and scalar role is not rejected by this new check, though existing single-scenario policy rules may still later mark multi-scenario reports ineligible if applicable.
* Multi-scenario manifest with derived roles `[base, stress]` remains valid.
* Legacy `cost_model` manifest remains valid.

### P1: Add full `run_research_backtest()` integration coverage

Add a new integration test file if one does not already exist, for example:

```text
tests/test_research_backtest_integration.py
```

Use synthetic SQLite candles. Do not require network, real exchange data, or live credentials.

The test should:

1. create a temporary SQLite database with the `candles` table schema expected by `load_dataset_split`;
2. insert enough candles for train, validation, and final_holdout date ranges;
3. build a manifest with:

   * one parameter candidate;
   * final holdout present;
   * execution model generating at least two scenarios;
   * `scenario_policy="must_pass_base_and_survive_stress"`;

   * one stress scenario that deterministically fails, using something reliable such as `order_failure_rate=[0.0, 1.0]` or another existing deterministic failure mechanism;

   * the candidate has `scenario_results`;

   * top-level `acceptance_gate_result == "FAIL"`;
   * `scenario_fail_count > 0`;
   * `gate_fail_reasons` includes a scenario-policy failure;
   * `candidate_profile_hash` exists;
   * scenario result objects include `scenario_role` and `scenario_role_source`;
6. write the report through existing report writer behavior if `run_research_backtest()` already does so;
7. call `promote_candidate()` and assert it refuses promotion with a scenario-policy reason.

Also add a positive integration case if practical:

* both base and stress scenarios pass;
* final holdout evidence exists;
* candidate top-level gate is `PASS`;
* promotion succeeds.

If a positive case is too brittle due to SMA signal behavior, document why and keep the negative integration case.
