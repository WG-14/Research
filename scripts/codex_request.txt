## Mission

Make scenario policy semantics explicit, predictable, and test-locked.

This patch is not about adding new trading features. It is about removing ambiguity from the research-validation and promotion pipeline so that future strategy candidates are evaluated and promoted under clear, auditable, machine-checkable policy rules.

The system must not silently infer unsafe policy behavior, must not confuse a single execution scenario with a base-plus-stress scenario suite, and must not hide optional calibration warnings inside a passing promotion artifact.

## Hypotheses to verify or falsify

Before coding, inspect the repository and classify each hypothesis as confirmed, partially confirmed, or rejected.

### Hypothesis 1: single-scenario `execution_model` manifests may default to an overly strict scenario policy

Expected current risk:

* `_parse_execution_model()` may default `scenario_policy` to `must_pass_base_and_survive_stress` whenever `execution_model` is present.
* A manifest with exactly one execution scenario may therefore require both base and stress evidence even though only one scenario exists.
* This can cause confusing candidate failures for otherwise valid single-scenario research runs.

Concrete example to test:

```json
{
  "execution_model": {
    "type": "fixed_bps",
    "fee_rate": [0.0004],
    "slippage_bps": [10]
  }
}
```

Expected safe behavior:

* Either default to `single_scenario` when exactly one scenario is generated;
* Or require explicit `scenario_policy` whenever `execution_model` is supplied;
* But do not silently apply `must_pass_base_and_survive_stress` to a single generated scenario.

### Hypothesis 2: base/stress scenario role assignment is still implicit and can become ambiguous

Expected current risk:

* Reports contain `scenario_role`, and `_apply_scenario_policy()` uses `base` vs `stress`.
* But `ExecutionScenario` may not have a first-class `scenario_role` field.
* Role assignment may be derived internally from scenario index or heuristic cost assumptions.
* This becomes ambiguous when manifests generate multiple scenarios from arrays such as:

```text
slippage_bps=[5, 10, 20]
latency_ms=[0, 1000]
partial_fill_rate=[0.0, 0.05]
```

Expected safe behavior:

* Scenario role should be explicit in the schema when supplied.
* If role is derived, the report should clearly state that it was derived and how.
* `must_pass_base_and_survive_stress` should require at least one base scenario and at least one stress scenario.
* Ambiguous role assignment should not silently pass.

### Hypothesis 3: optional `calibration_strictness="warn"` breaches may be too quiet in promotion artifacts

Expected current risk:

* Optional calibration warning mode may allow promotion even when the calibration gate is `FAIL`.
* That may be acceptable by policy, but the promotion artifact may only show `gate_result = PASS`, making the warning easy for an operator to miss.

Expected safe behavior:

* Promotion may proceed in warning mode, but the artifact must clearly expose:

  * `has_execution_calibration_warning`;
  * `execution_calibration_warning_reasons`;
  * `promotion_warnings`.
* CLI/report summaries should also surface warning counts or warning reasons where practical.

### Hypothesis 4: scenario-policy tests are still too unit-level and do not fully protect the actual pipeline

Expected current risk:

* Tests may directly exercise `_apply_scenario_policy()` and `_report_payload()`.
* But they may not run the actual `run_research_backtest()` path with:

  * manifest parsing;
  * execution scenario generation;
  * candidate aggregation;
  * scenario role assignment;
  * candidate profile hash generation;
  * report writer payload structure;
  * promotion refusal behavior.

Expected safe behavior:

* Add at least one full integration test using synthetic SQLite candles and a manifest.
* The test should prove that `run_research_backtest()` emits one aggregated candidate per parameter candidate, with multiple `scenario_results`, correct top-level gate result, and promotion refusal when required scenario evidence fails.

## Required patch priorities

### P1: Fix single-scenario scenario-policy default

Implement one of the following safe designs.

Preferred design:

* Generate scenarios first.
* If `execution_model.scenario_policy` is absent:

  * use `single_scenario` when exactly one scenario is generated;
  * use `must_pass_base_and_survive_stress` when multiple scenarios are generated.
* Preserve `legacy_cost_model_single_pass` for legacy `cost_model`-only manifests.

Alternative acceptable design:

* If `execution_model` is supplied and `scenario_policy` is absent, raise `ManifestValidationError`.
* Error text must clearly say that `execution_model.scenario_policy` is required to avoid ambiguous validation policy.

Do not keep the current behavior if it silently treats a single generated scenario as `must_pass_base_and_survive_stress`.

Tests required:

* Single generated execution scenario with no explicit policy parses as `single_scenario`, or fails with a clear manifest validation error if you choose explicit-only policy.
* Multiple generated execution scenarios with no explicit policy parse as `must_pass_base_and_survive_stress`, or fail with a clear manifest validation error if explicit-only policy is chosen.
* Legacy `cost_model` manifests still parse as `legacy_cost_model_single_pass`.

### P1: Add first-class `scenario_role` support

Patch the manifest/schema layer so `ExecutionScenario` can carry:

```text
scenario_role: "base" | "stress"
scenario_role_source: "manifest" | "derived"
```

Required behavior:

* `execution_model.scenario_role` may be optional in manifests.
* If supplied, validate it strictly as `base` or `stress`.
* If not supplied, preserve current derived behavior, but record `scenario_role_source = "derived"`.
* If supplied, record `scenario_role_source = "manifest"`.
* Scenario payloads and reports must include both `scenario_role` and `scenario_role_source`.
* `_apply_scenario_policy()` must use the scenario result role, not an undocumented heuristic.

For generated scenario products:

* If the manifest supplies a scalar `scenario_role`, apply it to all generated scenarios.
* If the manifest supplies role arrays or per-scenario roles, implement only if it fits cleanly. Otherwise do not add a half-supported schema.
* If multiple generated scenarios all have role `base` and policy is `must_pass_base_and_survive_stress`, the candidate must fail because no stress scenario exists.
* If multiple generated scenarios all have role `stress` and policy is `must_pass_base_and_survive_stress`, the candidate must fail because no base scenario exists.

Tests required:

* Manifest-supplied `scenario_role = "base"` appears in scenario reports with `scenario_role_source = "manifest"`.
* Missing scenario role is derived and reported as `scenario_role_source = "derived"`.
* `must_pass_base_and_survive_stress` fails when no base role exists.
* `must_pass_base_and_survive_stress` fails when no stress role exists.
* Unsupported role values fail manifest validation.

### P1/P2: Surface optional calibration warnings in promotion artifacts

Patch promotion artifact creation so optional warn-mode calibration breaches are visible at top level.

Required behavior:

* If `execution_calibration_required=false`, `execution_calibration_strictness="warn"`, and `execution_calibration_gate.status == "FAIL"`:

  * promotion may still pass;
  * artifact must contain `has_execution_calibration_warning = true`;
  * artifact must contain `execution_calibration_warning_reasons`;
  * artifact must include those reasons in `promotion_warnings`.
* If no warn-mode calibration breach exists:

  * `has_execution_calibration_warning = false`;
  * `execution_calibration_warning_reasons = []`;
  * `promotion_warnings` should still exist, even if empty.
* Do not downgrade required calibration failures into warnings.
* Do not allow `calibration_required=true` failures to promote.

Tests required:

* Optional warn calibration breach promotes but records top-level warning fields.
* Required calibration breach still refuses promotion.
* No calibration breach produces empty warning fields.
* Candidate profile hash and promotion artifact hash include the warning evidence where appropriate.

## Recovery and safety requirements

All ambiguous policy behavior must fail safely or be made explicit.

If scenario policy is unsupported, fail manifest validation or mark candidates ineligible with `scenario_policy_unsupported`.

If `must_pass_base_and_survive_stress` has no base scenario, fail with:

```text
scenario_policy_no_passing_base_scenario
```

If `must_pass_base_and_survive_stress` has no stress scenario, fail with:

```text
scenario_policy_no_passing_stress_scenario
```

If scenario role is invalid, fail manifest validation.

If optional calibration warning mode is used, promotion may proceed only if all hard gates pass, and the warning must be recorded.

If required calibration fails, promotion must remain refused.

If final holdout is required and missing, promotion must remain refused.

Do not delete existing research artifacts during normal runs.

Do not overwrite reports except through existing safe atomic report-writing conventions.
