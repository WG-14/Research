## Hypotheses to verify or refute

### H1: `MetricContractV2` may emit non-finite numeric values into JSON artifacts

Specifically verify whether `profit_factor=float("inf")`, `NaN`, or any other non-finite float can reach:

* `metrics_v2.as_dict()`
* research reports
* candidate profiles
* promotion artifacts
* lineage or reproducibility hashing

This is a P0 issue if true. Python’s default `json.dumps()` may serialize `Infinity`, but that is not strict JSON. Promotion-grade research artifacts must be strict JSON-compatible for external validation, deterministic hashing, and long-term preservation.

### H2: Metrics gate policy is not sufficiently bound into candidate/profile/promotion evidence

Verify whether the actual metrics policy used for the candidate is first-class evidence, not merely the resulting `metrics_v2` values.

The applied thresholds and flags must be inspectable and hash-bound:

* `min_cagr_pct`
* `min_expectancy_per_trade_krw`
* `min_expectancy_per_trade_pct`
* `max_exposure_time_pct`
* `max_avg_holding_time_minutes`
* `max_fee_drag_ratio`
* `max_slippage_drag_ratio`
* `reject_open_position_at_end`
* `metrics_contract_required`

If these are only in the manifest but not clearly bound into the candidate profile / promotion artifact, patch it.

### H3: Failed or skipped execution branches may skip equity-curve and MDD marking

Inspect `run_sma_backtest()` branches where execution reference is missing, fill fails, or execution is skipped. If those branches append a trade and `continue` before the common equity/MDD mark, then a candle can be omitted from the equity curve and risk metrics can be understated.

This is critical when:

* an exit SELL fails while a position is open
* missing quote skips an execution
* stress execution failure occurs
* a large adverse candle happens during the failed/skipped execution path

### H4: Partial SELL residual-position behavior may not be integration-tested

The metrics calculator may test partial/failure/skipped execution counts, but verify whether an end-to-end SMA backtest with partial SELL leaves the residual position correctly represented.

The integration behavior must be pinned by tests:

* `open_position_at_end=true`
* `unrealized_pnl_end` is based on the residual position
* closed trade records include only realized closed quantity/PnL
* holding interval is not falsely treated as fully closed
* exposure time includes the still-open interval through period end

### H5: Promotion artifact observability may still require opening nested candidate profile internals

Operators should be able to open a promotion artifact and immediately see metrics contract evidence and applied metrics gate policy. If `metrics_v2` and policy details are only buried inside `candidate_profile`, improve top-level promotion artifact observability.

## Priorities

Implement in this order.

### P0 — Eliminate non-finite JSON from metrics, reports, hashes, and promotion artifacts

No research/promotion/report artifact should contain `Infinity`, `-Infinity`, or `NaN`.

Preferred approaches, in order:

1. Make `MetricContractV2` emit finite numbers or `None` only.
2. For unbounded profit factor, use:

   * `profit_factor: null`
   * `profit_factor_unbounded: true`
   * `limitation_reasons` containing a stable code such as `profit_factor_unbounded_no_losses`
3. Add a strict JSON sanitizer or validation function at artifact boundaries so future non-finite values fail early.
4. Update `hashing.canonical_json_bytes()` and/or artifact writing paths if needed so non-finite payloads cannot silently become non-standard JSON.

Do not merely rely on Python’s permissive `json.dumps()` behavior.

Add tests that prove:

```python
json.dumps(payload, allow_nan=False)
```

passes for:

* `metrics_v2.as_dict()`
* a research backtest report payload
* candidate profile payload
* promotion artifact payload where feasible

Also test an all-winning closed-trade sample to ensure no `Infinity` reaches serialized artifacts.

### P1 — Bind metrics gate policy into candidate/profile/promotion evidence

Add a deterministic metrics gate policy payload, for example:

```text

  metrics_schema_version
  min_cagr_pct

  max_fee_drag_ratio
  max_slippage_drag_ratio
  reject_open_position_at_end
  metrics_contract_required
metrics_gate_policy_hash
```

Implementation requirements:

* Build this from `manifest.acceptance_gate`, not from ad hoc candidate state.
* Attach it to each candidate payload.
* Attach it to each scenario result if scenario-level evaluation needs it, or clearly document candidate-level binding.
* Include it in `build_candidate_profile()`.
* Include it in promotion artifact top-level fields.
* Include `metrics_schema_version`, `validation_metrics_v2`, and `final_holdout_metrics_v2` in promotion artifact top-level fields, or add a compact `metrics_v2_summary` plus full candidate profile reference if full payload size is a concern.
* If `metrics_contract_required=true` and `validation_metrics_v2` is missing or schema mismatched, promotion validation must fail closed with stable reason codes.
* If final holdout is required and present, enforce the same contract for `final_holdout_metrics_v2`.

Add tests that prove:

* changing only a metrics threshold changes the manifest hash and candidate/profile evidence hash
* `metrics_contract_required=true` with removed or malformed `validation_metrics_v2` refuses promotion
* promotion artifact exposes the applied metrics gate policy without requiring deep candidate-profile inspection
* policy hash is deterministic

Use stable reason codes such as:

```text

metrics_gate_policy_hash_missing
metrics_gate_policy_hash_mismatch
metrics_v2_missing
metrics_contract_missing
validation_metrics_v2_missing
final_holdout_metrics_v2_missing
```

Adapt to existing naming conventions, but keep the meaning explicit.
