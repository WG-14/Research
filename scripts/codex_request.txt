2. Hypothesis A: `research-reproduce` trusts report `content_hash`

Verify whether `reproduce_promotion()` or its helper, likely `_verify_artifact_hash()`, opens backtest/walk-forward reports and does something equivalent to:

```python
actual = str(payload.get("content_hash") or "")
```

If yes, fix it.

Required design:
`research-reproduce` must not trust the artifact’s embedded `content_hash` for backtest/walk-forward reports. It must recompute the report’s canonical content hash from the loaded artifact body, excluding the `content_hash` field, using the repo’s deterministic hash conventions.

Preferred implementation:

```python
actual = sha256_prefixed(
    content_hash_payload({k: v for k, v in payload.items() if k != "content_hash"})
)
```

If the repo has a report-writer-specific official hash helper, use that instead. Do not create a conflicting hash convention.

Required failure behavior:

* If the report body is changed but the old `content_hash` field remains, `research-reproduce` must fail.
* Backtest report tamper should report `backtest_report_hash_mismatch`.
* Walk-forward report tamper should report `walk_forward_report_hash_mismatch`.
* The structured output should include the expected and actual hashes in `mismatches`.

Required tests:

* valid promotion + valid lineage + valid report artifacts → `research-reproduce` returns `ok=true`
* backtest report body tampered, old `content_hash` left unchanged → `ok=false`, reason `backtest_report_hash_mismatch`
* walk-forward report body tampered, old `content_hash` left unchanged → `ok=false`, reason `walk_forward_report_hash_mismatch`
* embedded report `content_hash` tampered while body unchanged → fail clearly
* missing report file still produces the existing missing-artifact reason

3. Hypothesis B: promotion silently allows no-lineage reports

Verify whether `promote_candidate()` does this:

* read `report.get("lineage")`
* if lineage exists, validate it
* if lineage is missing, create promotion artifact with:

  * `lineage_required=false`
  * `legacy_compatibility_used=true`
  * no hard failure

If yes, fix it.

Required design:
New promotion must be lineage-backed by default. Legacy/no-lineage promotion is allowed only with explicit operator intent.

Implement:

```python
promote_candidate(
    *,
    experiment_id: str,
    candidate_id: str,
    manager: PathManager,
    generated_at: str | None = None,
    allow_legacy_lineage: bool = False,
) -> PromotionResult:
```

Default behavior:

* If `backtest_report.json` has no valid `lineage`, promotion must fail with:

  * `promotion refused: lineage_missing`

Explicit compatibility behavior:

* If `allow_legacy_lineage=True`, no-lineage promotion may proceed.
* The artifact must include:

  * `legacy_compatibility_used=true`
  * `lineage_required=false`
  * `lineage_hash=None`
  * a clear warning such as `legacy_lineage_compatibility_used`
* CLI output must print that warning.
* Audit/verification output must continue to expose `legacy_compatibility_used`.

CLI wiring:
Add an explicit CLI flag:

```bash
uv run bithumb-bot research-promote-candidate \
  --experiment-id <id> \
  --candidate-id <id> \
  --allow-legacy-lineage
```

Do not make this the default. Do not silently enable it based on environment variables.

Required tests:

* report with valid lineage → promotion passes
* report without lineage → promotion fails by default with `lineage_missing`
* report without lineage + explicit `allow_legacy_lineage=True` → promotion passes and artifact has `legacy_compatibility_used=true`
* CLI passes the flag through correctly
* CLI without the flag fails on no-lineage report
* invalid lineage hash still fails closed

4. Hypothesis C: decision-equivalence is not mandatory transition evidence

* profile promotion CLI paths
* tests for missing or failed decision-equivalence reports

If decision-equivalence is already mandatory and well-tested, document the evidence and only add missing edge tests. If it is not mandatory, fix it.

Required design:
Decision-equivalence must be mandatory for profile transition evidence where the system is trying to prove research/paper/runtime continuity.

At minimum:

* `paper_validation_evidence` must include:

  * `decision_equivalence_report_path`
  * `decision_equivalence_content_hash`
* `live_readiness_evidence` must either also require the same fields, or the code must define an explicit policy explaining why only paper validation requires it. Prefer requiring it for both unless existing design clearly says otherwise.

Decision-equivalence validation must fail closed when:

* path missing
* hash missing
* report file missing
* report hash mismatch
* report `ok` is false
* `mismatched_decision_count > 0`
* `missing_research_decisions` is non-empty
* `missing_runtime_decisions` is non-empty
* profile hash mismatch
* market mismatch
* interval mismatch
* data/db fingerprint mismatch, if both evidence and report provide comparable fingerprints

Hash rule:
Do not trust the report’s embedded `content_hash`. Recompute it from canonical report body excluding `content_hash`, using the repo’s deterministic hash convention.

Required tests:

* paper validation evidence missing decision-equivalence fields → fail
* live readiness evidence missing decision-equivalence fields → fail, unless explicitly documented otherwise
* decision-equivalence report path missing → fail
* decision-equivalence hash mismatch → fail
* decision-equivalence `ok=false` → fail
* `mismatched_decision_count > 0` → fail
* non-empty `missing_research_decisions` → fail
* non-empty `missing_runtime_decisions` → fail
* profile hash mismatch → fail
* market mismatch → fail
* interval mismatch → fail
* valid decision-equivalence report → profile transition evidence passes
