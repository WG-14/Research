Core concern:

The repository already has a strong research evidence pipeline, but the market data contract is not yet fully closed around data sufficiency, dataset quality, execution feasibility, and reproducible promotion evidence. The immediate issue is not merely “the research dataset uses candles.” The deeper issue may be that the chain:

market data source → dataset quality proof → execution model calibration → research promotion → approved profile / operational readiness

is still partly optional, partly documented, or partly disconnected in code.

Do not dilute this problem into generic “add better data validation.” Keep the priorities sharp:

1. First prove whether the existing candle dataset is research-grade.
2. Then bind execution-quality calibration into research/promotion evidence.
3. Then add top-of-book snapshots as a second research data layer.
4. Then make strategy/data-source requirements explicit so unsupported strategies fail closed.
5. Preserve reproducibility, observability, and recovery safety.

Important starting observations to verify, not blindly assume:

* The research manifest currently appears to require `dataset.source=sqlite_candles`.
* Candle snapshots appear to include only `ts, pair, interval, open, high, low, close, volume`.
* `DatasetSnapshot.content_hash()` appears to prove payload identity but not dataset suitability.
* The candle table primary key prevents duplicate keys but does not prove full expected coverage.
* The backtest path appears to use candle close as the reference price and does not appear to populate `best_bid`, `best_ask`, or `spread_bps` in `ExecutionRequest`.
* `ExecutionRequest` / `ExecutionFill` already have fields for quote/spread/depth-related metadata.
* `marketdata.py` already has public orderbook top fetch helpers.
* `execution_quality.py` and `execution_calibration.py` already provide a path from real fills to calibration metrics, but calibration may not be mandatory for all research promotion paths.
* Documentation acknowledges that candle datasets do not contain orderbook depth or intra-candle path data.

Maintain hypotheses as hypotheses. For each hypothesis below, either verify, refute, or mark as partially supported with exact code evidence.

Hypothesis 1:
The candle-centered research data contract causes the system to overestimate what kinds of strategy edges can be validated. SMA-style directional/filter research may be acceptable with candles, but spread-sensitive, latency-sensitive, partial-fill-sensitive, microstructure, or intra-candle path-dependent strategies are not adequately validated.

How to verify or refute:
Compare the current research execution path with the data required to model spread, fill latency, partial fill, unfilled orders, and quote availability. Check whether any research strategy can currently require top-of-book or richer data. Check whether `ExecutionRequest.best_bid`, `best_ask`, and `spread_bps` are populated in research backtests. If they are not, confirm whether this is intentional and documented.

Hypothesis 2:
`DatasetSnapshot.content_hash()` proves reproducibility of a candle payload but not research suitability. The system may need a separate `DatasetQualityReport` artifact that proves coverage, interval consistency, missing buckets, OHLC invariants, and schema/source fingerprint.

How to verify or refute:
Inspect dataset loading and report generation. Determine whether current reports include expected candle count, actual candle count, missing bucket count/ranges, OHLC violations, non-positive prices, negative volume, interval mismatch, split coverage percentage, DB schema fingerprint, and dataset quality hash. If not present, implement it.

Hypothesis 3:
Execution-quality calibration exists but may not be a hard promotion contract. If `execution_model.calibration_required=true` is optional or not consistently enforced, candidates can be promoted using arbitrary fixed-bps assumptions.

How to verify or refute:
Inspect manifest parsing, research report generation, promotion, reproduction, and calibration comparison logic. Confirm whether calibration artifact content hash, market/interval binding, p90/p95 slippage, p95 latency, partial-fill rate, and model-breach rate are enforced before promotion. If enforcement exists, document exactly where. If gaps exist, patch them fail-closed.

Hypothesis 4:
The ingestion path may be optimized for recent candle upsert, not durable long-range research dataset construction. Long train/validation/final_holdout splits may therefore be under-covered unless an explicit coverage/backfill process exists.

How to verify or refute:
Inspect `cmd_sync`, any backfill commands, dataset loading, and tests. Determine whether long historical ranges can be constructed and verified. Do not implement a full historical backfill unless necessary for the current scoped fix, but ensure dataset quality reports expose insufficient range coverage clearly.

Hypothesis 5:
The research/runtime separation is safety-positive, but the research data layer is still effectively single-source. A typed market-data contract may be needed so strategies can declare required vs optional market data, and unsupported research configurations fail closed.

How to verify or refute:
Inspect the strategy registry, manifest schema, and research engine. Determine whether a strategy can declare required data sources. If not, implement the minimal contract needed to prevent future misuse without breaking current candle-based SMA research.

Implementation priority:

P0 — Must implement if missing:
Add a `DatasetQualityReport` mechanism for SQLite candle research datasets.

Required behavior:

* Produce a deterministic JSON-compatible payload for each dataset split.
* Include at least:

  * `schema_version`
  * `artifact_type = "dataset_quality_report"`
  * `source`
  * `market`
  * `interval`
  * `snapshot_id`
  * `split_name`
  * `start_ts`
  * `end_ts`
  * `expected_candle_count`
  * `actual_candle_count`
  * `coverage_pct`
  * `missing_bucket_count`
  * compact missing bucket ranges or a bounded sample plus total count
  * `duplicate_key_count`
  * `non_monotonic_ts_count`
  * `interval_mismatch_count` or equivalent interval consistency metric
  * `ohlc_violation_count`, using `low <= open <= high`, `low <= close <= high`, and `low <= high`
  * `non_positive_price_count`
  * `negative_volume_count`
  * `first_ts`
  * `last_ts`
  * `db_schema_fingerprint`
  * `dataset_content_hash`
  * `quality_gate_status`
  * `quality_gate_reasons`
  * deterministic `content_hash`, excluding volatile timestamps
* Compute expected candle count from manifest date range and interval.
* Support the repository’s current interval conventions. If interval parsing is incomplete, implement the minimum safe parser and fail closed for unsupported intervals.
* Do not silently pass unknown interval formats.
* Integrate the report into research output lineage/reporting so dataset suitability is visible and hash-bound.
* Make promotion fail closed, or at minimum research gate fail closed, when dataset quality is materially insufficient.
* Preserve backwards compatibility where appropriate, but do not let compatibility hide quality failures for new research artifacts.

Suggested gate policy:

* Missing report: fail for new-generation research promotion.
* Coverage below a strict threshold: fail.
* Any OHLC invariant violation: fail.
* Any non-positive price: fail.
* Negative volume: fail.
* Duplicate primary keys should normally be impossible at DB level, but if detected through raw diagnostics, fail or warn with explicit reason.
* Missing candles: fail unless a clearly named policy allows warning mode for smoke-only diagnostics. Do not allow warning-mode output to become promotion evidence.
