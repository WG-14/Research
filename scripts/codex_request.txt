## Mission

Prioritize design changes that prevent an entire class of false-positive backtest and promotion errors over isolated feature additions.

1. clear separation between smoke-test backtests and official research validation;

## Required design changes

Implement the highest-value subset that can be completed cleanly in this repository without overreaching. Prefer a coherent minimal architecture over many shallow options.

### 1. Demote root `backtest.py` to explicit smoke-test status

Make it impossible to confuse root `backtest.py` with the official research-validation path.

Acceptable implementations include one or more of:

* move it to `scripts/legacy_sma_smoke_backtest.py` or `scripts/smoke_backtest.py`;
* leave a thin compatibility wrapper at root that prints a strong warning;
* update README/docs/runbooks to state that root/simple backtests are not valid evidence for promotion, approval, or live readiness;
* add tests that assert the official validation path is the research CLI, not root `backtest.py`;
* ensure this script cannot produce candidate, promotion, or approved-profile artifacts.

The warning should be unambiguous:

> This is a smoke backtest only. It must not be used as evidence for strategy promotion, approved profiles, live readiness, or capital allocation.

### 2. Introduce a pluggable execution-model interface for research backtests

Refactor research backtesting so fill simulation is not hard-coded inside the strategy/backtest loop.

Create a small, testable execution-model abstraction, for example:

```text
src/bithumb_bot/research/execution_model/
  __init__.py
  base.py
  fixed_bps.py
  stress.py
```

At minimum, support:

* a fixed bps model equivalent to current behavior, for backwards compatibility;
* a stress model that can apply configurable slippage bps, latency assumptions, partial-fill rate, order-failure rate, and optional market-order extra cost;
* deterministic behavior when a seed is supplied.

Each simulated trade should record enough metadata to audit the execution assumption:

```text
signal_ts
decision_ts
submit_ts_assumption
side
order_type
reference_price
requested_qty
filled_qty
remaining_qty
avg_fill_price
fee
slippage_bps
latency_ms
fill_status
model_name
model_version
model_params_hash
```

Include optional fields if the repository has data for them:

```text
best_bid
best_ask
spread_bps
orderbook_depth_ref
intra_candle_policy
```

Do not fake unavailable orderbook data. If bid/ask/depth data is unavailable, record `null`/omitted fields and make the limitation explicit in reports.

### 3. Extend research manifests with execution scenarios

Add or support manifest configuration that separates old static cost fields from richer execution-model scenarios.

Example target shape:

```json
{
  "execution_model": {
    "type": "stress",
    "fee_rate": [0.0025],
    "slippage_bps": [5, 10, 20, 40],
    "latency_ms": [0, 500, 1500, 3000],
    "partial_fill_rate": [0.0, 0.03, 0.10],
    "order_failure_rate": [0.0, 0.01],
    "market_order_extra_cost_bps": [0, 5, 15],
    "scenario_policy": "must_pass_base_and_survive_stress",
    "seed": 42
  }
}
```

Backward compatibility requirement:

* existing manifests using `cost_model.fee_rate` and `cost_model.slippage_bps` must still work;
* when legacy cost fields are used, normalize them internally into an execution-model scenario;
* emitted artifacts should clearly state whether the run used legacy cost fields or the new execution-model config.

### 5. Connect calibration artifacts to research backtests and promotion gates

Research backtests should optionally consume a cost-model calibration artifact.

Required behavior:

* If a calibration artifact is supplied, compare observed p90/p95 execution costs against the manifest’s assumed cost model.
* If observed costs are materially worse than assumptions, either fail the run or emit a machine-readable warning depending on the configured strictness.
* Promotion or approved-profile creation must fail closed when required evidence is missing or when observed execution quality breaches the assumed research cost model.
* If promotion logic already checks manifest hashes, candidate hashes, walk-forward evidence, or cost hashes, extend those checks rather than replacing them.

Suggested fail-closed conditions:

```text
walk_forward_required=true but no walk-forward report exists
final_holdout is missing for a promotion candidate
parameter stability is below threshold
regime-specific performance violates acceptance gates
execution-quality p90/p95 slippage exceeds research assumptions
model_breach_rate exceeds configured threshold
paper/live-dry-run observation sample count is below threshold
manifest/candidate/cost-model hash mismatch
```

Only implement conditions that can be supported by existing repository data and tests. For unsupported conditions, add explicit TODO documentation and schema fields so the future path is clear.

### 6. Add robustness checks where feasible

Add or improve tests and metrics for:

* rolling walk-forward validation;
* regime-level performance decomposition;
* cost sensitivity;
* deterministic stress execution with seed;
* trade-level execution metadata;

If bootstrap or Monte Carlo is not currently present, add a small deterministic robustness module only if it fits cleanly. Otherwise, document it as a missing validation layer and add a testable interface or placeholder schema, not a misleading half-implementation.
