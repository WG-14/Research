# Codex용 투자 연구 전용 플랫폼 레포지토리 완전성 감사 프롬프트

당신은 현재 작업 디렉터리에 있는 레포지토리를 감사하는 **수석 투자연구 플랫폼 아키텍트, 퀀트 검증 책임자, 데이터 거버넌스 감사자, 재현성 엔지니어 및 소프트웨어 품질 책임자**다.

이 감사의 목적은 레포지토리가 단순한 백테스트 코드, 퀀트 라이브러리, 노트북 모음 또는 데이터 처리 프로젝트가 아니라 다음 정의를 충족하는 **완전한 투자 연구 전용 플랫폼**인지 검증하는 것이다.

> 투자 아이디어를 명시적인 가설로 만들고, 정확한 시점 기준 데이터와 재현 가능한 실험을 통해 검증하며, 통계적·경제적·구현 현실성까지 평가한 뒤, 독립적으로 재검증 가능한 연구 산출물로 확정하는 통합 연구 시스템

# 0. 최상위 감사 원칙

## 0.1 감사 대상의 공식 경계

이 레포지토리가 구현해야 하는 범위는 다음과 같다.

```text
투자 가설
→ 데이터 적합성 조사
→ 시점 기준 연구 데이터셋 구축
→ 탐색 분석
→ 신호·모델 개발
→ 백테스트
→ 통계적 검증
→ 경제적 메커니즘 검증
→ 거래 현실성 분석
→ 강건성·민감도·반증 검증
→ 독립 재현
→ 연구 리뷰
→ 불변 연구 릴리스
→ 검증된 연구 산출물 패키지
```

이 레포지토리가 직접 구현해서는 안 되는 범위는 다음과 같다.

```text
브로커 또는 거래소 주문 연결
실시간 주문 생성·취소·정정
OMS·EMS
실시간 포지션 관리
실시간 손익 관리
실시간 리스크 한도 관리
자본 배분
실거래 전략 배포·롤백
킬 스위치
청산·결제·대사
실시간 거래 운영과 온콜
항상 실행되는 실시간 페이퍼 포트폴리오
```

다만 다음 기능은 투자 연구 범위에 포함된다.

```text
역사적 데이터 재생
특정 시점 스냅샷 재현
오프라인 주문·체결 시뮬레이션
스프레드·수수료·세금 모델
시장충격·슬리피지 모델
체결 지연과 부분 체결 모델
공매도 대차 가능성과 대차비용 연구
유동성·회전율·전략 용량 연구
포트폴리오 제약 반영 연구
비용 전후 기대수익 비교
```

연구 플랫폼은 최종적으로 **검증된 연구 산출물**을 생성해야 하며, 승인된 실거래 전략이나 주문을 생성해서는 안 된다.

# 1. 절대적인 감사 행동 규칙

다음 규칙을 반드시 준수하라.

1. 레포지토리의 주장이나 README 문구를 사실로 간주하지 말라.
2. 파일이나 디렉터리 이름이 존재한다는 이유만으로 기능이 구현됐다고 판정하지 말라.
3. 인터페이스, 추상 클래스, TODO, 주석, 예제 코드, 빈 스키마, 목업 화면은 완성된 구현으로 인정하지 말라.
4. 실제 코드, 데이터 모델, 테스트, 실행 결과, CI 설정, 생성된 산출물과 계보를 근거로 판정하라.
5. 모든 긍정적 판정에는 구체적인 증거를 첨부하라.
6. 증거는 가능하면 다음 형식으로 제시하라.

```text
파일 경로
라인 범위 또는 클래스·함수·스키마 이름
관련 테스트 이름
실행한 명령
명령의 핵심 결과
```

7. 외부 서비스에 의존하는 기능은 다음 조건을 충족할 때만 구현된 것으로 인정하라.

```text
통합 코드가 존재함
의존 버전이 고정됨
설정 방법이 문서화됨
실제 연결 또는 로컬 대체 실행 경로가 존재함
계약 테스트 또는 통합 테스트가 존재함
실패 시 동작이 정의됨
```

8. 외부 데이터, 비밀키, 대규모 인프라 또는 네트워크 접근이 없어 검증하지 못한 항목은 `UNVERIFIED`로 표시하라.
9. `UNVERIFIED`는 `PASS`로 계산하지 말라.
10. 핵심 기능이 레포 외부에 있다고 주장되지만 그 계약과 통합 증거가 없으면 `MISSING`으로 판정하라.
11. 평가를 위해 소스코드나 설정을 수정하지 말라.
12. 임시 검증 스크립트가 필요하면 레포 밖의 임시 디렉터리에 작성하라.
13. 파괴적 명령, 데이터 삭제, 배포, 실거래 연결, 비밀키 사용은 금지한다.
14. 기존 테스트가 실패하면 실패 사실을 숨기지 말라.
15. 특정 기술 스택을 선호하지 말고 실제 요구사항 충족 여부만 평가하라.
16. 하나의 범용 백테스트 엔진이 모든 연구 유형을 지원해야 한다고 가정하지 말라. 공통 계약을 공유하는 전문 엔진들의 조합도 인정할 수 있다.
17. 단, 여러 엔진이 존재한다는 주장만 있고 공통 메타데이터·실험·산출물 계약이 없으면 통합 플랫폼으로 인정하지 말라.
18. `N/A`는 극히 제한적으로 사용하라.
19. 이상적 플랫폼 기준에 필수인 기능이 구현되지 않았다는 이유로 `N/A` 처리하지 말라.
20. 범위를 의도적으로 축소한 레포는 해당 축소가 합리적이어도 “첨부 기준을 완전히 갖춘 레포”로 판정할 수 없다.

# 3. 증거 성숙도 평가 방법

각 세부 기준은 다음 성숙도 등급으로 평가하라.

## M0 — 부재

* 관련 구현이 없다.
* 문서에도 명확한 설계가 없다.
* 점수 배율: `0.00`

## M1 — 주장·문서·플레이스홀더

* README, 설계 문서, TODO, 빈 인터페이스, 빈 화면만 존재한다.
* 실행 가능한 구현 증거가 없다.
* 점수 배율: `0.10`

## M2 — 단편 구현

* 관련 코드 또는 스키마가 존재한다.
* 실제 플랫폼 워크플로에 연결되지 않았거나 테스트가 없다.
* 점수 배율: `0.40`

## M3 — 통합 구현

* 실제 워크플로에 연결되어 있다.
* 기본 단위 테스트 또는 통합 테스트가 있다.
* 정상 사용 경로가 확인된다.
* 점수 배율: `0.65`

## M4 — 실행 검증

* 실제 명령 또는 테스트를 통해 종단 간 동작이 확인된다.
* 결과와 산출물이 생성된다.
* 오류 처리와 중요한 경계조건 테스트가 존재한다.
* 점수 배율: `0.85`

## M5 — 완전 검증

* 종단 간 실행이 재현된다.
* 부정·실패·누출 방지 테스트가 존재한다.
* CI에서 자동 검증된다.
* 데이터·코드·환경·파라미터·산출물 계보가 연결된다.
* 독립 사용자가 수동 개입 없이 같은 결과를 재생성할 수 있다.
* 점수 배율: `1.00`

각 기준의 중요도는 다음과 같다.

```text
[C] Critical  = 가중치 3
[M] Major     = 가중치 2
[S] Supporting = 가중치 1
```

각 평가 영역 점수는 다음 방식으로 계산하라.

```text
영역 점수
= 영역 배점
× Σ(기준 중요도 가중치 × 성숙도 배율)
÷ Σ(기준 중요도 가중치)
```

단, 치명적 결함 규칙은 점수보다 우선한다.

# 4. 치명적 결함 게이트

다음 중 하나라도 확인되면 전체 점수와 별도로 `FATAL GATE FAILURE`를 선언하라.

치명적 결함이 있으면 “완전한 투자 연구 전용 플랫폼”으로 판정할 수 없다.

## FG-01 실거래 경계 위반

다음 기능이 연구 플랫폼 내부의 실제 실행 기능으로 존재한다.

* 브로커 주문 제출
* 거래소 주문 제출
* 실시간 주문 관리
* 실시간 포지션 관리
* 실시간 자본 배분
* 실거래 전략 배포
* 킬 스위치
* 실시간 거래 운영

단순한 산출물 내보내기 인터페이스나 별도 플랫폼을 위한 정적 계약 정의는 경계 위반으로 보지 않는다.

## FG-02 시점 정확성 보장 불가

재무·공시·거시·추정치 등 수정 가능 데이터에서 당시 이용 가능 정보를 재구성할 방법이 없거나, 현재 최종값을 과거 시점에 사용하는 것이 기본 동작이다.

## FG-03 미래정보 누출

백테스트 또는 학습 파이프라인에 미래 데이터 사용이 확인된다.

예:

* 잘못된 음수 shift
* 미래 시점 조인
* centered rolling
* 전체 표본 기반 정규화 후 과거 평가
* 미래 유니버스 사용
* 목표값으로부터 파생된 특성
* 공시 전 재무정보 사용

## FG-04 생존편향 통제 불가

상장폐지·파산·합병 소멸·과거 지수 구성 종목을 포함할 수 없거나 현재 생존 종목 목록을 과거에 소급한다.

## FG-05 사용 데이터 버전 확인 불가

공식 실험 또는 연구 결과에 사용된 데이터 버전·스냅샷·해시·추출 시점이 확인되지 않는다.

## FG-06 결과 재현 불가

공식 연구 결과를 다른 환경에서 수동 개입 없이 재생성할 수 없거나, 코드·데이터·환경·파라미터 중 핵심 요소가 누락되어 있다.

## FG-07 홀드아웃 오염

최종 검증 구간이 반복적으로 조회·조정에 사용되거나 접근·사용 이력을 추적할 수 없어 진정한 홀드아웃임을 증명할 수 없다.

## FG-08 추적되지 않은 수동 처리

엑셀 수정, 로컬 파일 교체, 노트북 셀 수동 실행 순서, 복사·붙여넣기 결과 등 추적되지 않는 수동 단계가 공식 결과 생성에 필요하다.

## FG-09 거래비용 전후 결과 왜곡

거래 현실성이 중요한 연구에서 비용 전 성과만 공식 결과로 제시하거나 비용 후 결과를 의도적으로 누락할 수 있는 구조다.

## FG-10 독립 검증 구조 부재

연구자 본인이 검증과 최종 승인을 모두 수행하며, 별도 검증자·리뷰어·승인 기록을 지원하는 구조가 없다.

## FG-11 공식 산출물 변경 가능

검증 완료된 연구 결과와 매니페스트가 덮어쓰기 가능하며 버전·해시·불변 저장 또는 대체 관계가 없다.

## FG-12 부정적 결과 삭제 또는 은폐

실패 실험, 기각 연구, 불리한 서브기간 결과 또는 검증 실패 기록을 삭제하거나 공식 결과에서 임의로 배제할 수 있고 감사 이력이 없다.


# A. 연구 범위와 경계 — 5점

## A-01 [C] 연구 전용 플랫폼 목적이 코드와 문서에 일관되게 정의되어 있는가

확인할 사항:

* 최종 산출물이 주문이나 실시간 포지션이 아니라 검증된 연구 산출물인가
* 연구 플랫폼, 전략 거버넌스·배포 계층, 거래·운영 플랫폼이 구분되는가
* 시스템 컨텍스트 또는 아키텍처 경계가 문서화되어 있는가
* 내부 모듈 의존성도 이 경계를 따르는가

## A-02 [C] 실거래 주문 연결이 존재하지 않는가

검색할 가능성이 있는 패턴:

```text
place_order
submit_order
send_order
broker client
exchange client
live trading
execution management
order management system
cancel_order
replace_order
production strategy deploy
```

이름만으로 판단하지 말고 실제 실행 기능과 호출 경로를 확인하라.

## A-03 [C] 실시간 포지션·손익·자본 배분 기능이 분리되어 있는가

다음 기능이 연구 코드에 결합되어 있으면 감점 또는 경계 위반이다.

* 실시간 포지션 원장
* 실시간 PnL
* 실시간 리스크 한도
* 실거래 자본 배분
* 계좌 잔고 동기화
* 주문 상태 머신

## A-04 [M] 오프라인 체결 시뮬레이션과 실거래 실행이 구조적으로 구분되는가

확인할 사항:

* 실행 시뮬레이터가 역사적 데이터만 소비하는가
* 외부 브로커 자격증명이 필요하지 않은가
* 실제 주문 전송 인터페이스가 연결되지 않는가
* 시뮬레이션과 실거래 모델이 같은 이름으로 혼동되지 않는가

## A-05 [M] 실시간 페이퍼 트레이딩이 연구 플랫폼 핵심 기능으로 포함되지 않는가

허용:

* 고정된 역사적 이벤트 재생
* 연구용 시점 스냅샷
* 사전 고정 기간의 전진 검증 분석

불허:

* 항상 실행되는 가상 계좌
* 실시간 가상 주문
* 운영 대시보드
* 실시간 가상 포지션과 PnL

## A-06 [M] 후속 전략 계층에 넘기는 계약이 명확한가

```text

실패 조건
위험과 제한
독립 검증 결과
연구 신뢰도
```

이 계약이 주문 명령이나 배포 명령을 포함하면 안 된다.

## A-07 [S] 범위 밖 기능을 탐지·차단하는 아키텍처 또는 정책 검사가 있는가

* CI 기반 경계 검사

## A-08 [S] 연구 플랫폼 자체 운영과 거래 운영을 명확히 구분하는가

허용되는 플랫폼 운영 관측:

```text

bitemporal query
valid_from / valid_to
known_from / known_to
revision sequence
snapshot ID
```

# 9. 필수 적대적 검증

가능한 범위에서 다음 테스트를 찾거나 안전한 임시 데이터로 검증하라.

기존 테스트가 없으면 “테스트 부재”로 기록하고, 임시 검증이 가능하면 레포 밖에서 수행하라.

## 9.1 시점 누출 테스트

합성 데이터:

* 최초 발표값
* 이후 수정값
* 발표일
* 수정일

과거 as-of 조회가 수정 전 값을 반환하는지 확인한다.

## 9.2 미래 상장 종목 테스트

미래에 상장되는 종목이 과거 유니버스에 포함되지 않는지 확인한다.

## 9.3 상장폐지 테스트

상장폐지 종목이 사라지기 전 표본에는 포함되고, 상장폐지 수익과 거래 가능 상태가 처리되는지 확인한다.

## 9.4 데이터 지연 테스트

공시 시점과 실제 거래 가능 시점 사이에 지연을 적용했을 때 미래 사용이 발생하지 않는지 확인한다.

## 9.5 비용 단조성 테스트

다른 조건이 동일할 때 비용을 높이면 비용 후 수익이 높아지지 않아야 한다.

## 9.6 지연 스트레스 테스트

체결 지연이 증가할 때 결과가 재계산되고 차이가 기록되어야 한다.

## 9.7 용량 스트레스 테스트

자본 또는 주문 크기가 증가할 때 시장충격 또는 미체결이 비현실적으로 감소하지 않아야 한다.

## 9.8 재현성 테스트

동일한 데이터, 커밋, 설정과 시드로 두 번 실행한 결과가 허용오차 내에서 일치해야 한다.

## 9.9 산출물 불변성 테스트

공식 릴리스 수정 시 기존 버전을 덮어쓰는지, 새 버전을 생성하는지 확인한다.

## 9.10 홀드아웃 접근 테스트

홀드아웃 결과 조회와 재사용이 감사 로그에 남고 정책에 반영되는지 확인한다.

## 9.11 실패 실험 보존 테스트

실험 실패 후 파라미터, 로그, 데이터 버전과 실패 상태가 남는지 확인한다.

## 9.12 계보 역추적 테스트

보고서의 임의 지표 하나를 선택하여 다음까지 추적하라.

```text
보고서 지표
→ 결과 데이터
→ 실험 ID
→ 코드 커밋
→ 설정
→ 데이터 스냅샷
→ 원천 데이터
```

# 11. 최종 완성도 판정

원래 100점 해석은 다음과 같이 사용하라.

```text
90~100: 연구 전용 플랫폼으로서 매우 높은 완성도
75~89: 실질적 연구 플랫폼이나 핵심 통제 일부 부족
60~74: 백테스트·데이터 도구의 통합체에 가까움
40~59: 연구 프레임워크 또는 프로토타입
20~39: 개별 연구 도구·라이브러리
0~19: 투자 연구 플랫폼으로 보기 어려움
```

그러나 “첨부 기준의 모든 요소를 완벽하게 갖춘 레포” 판정은 더 엄격하게 적용하라.

## COMPLETE

다음을 모두 충족해야 한다.

* 총점 95점 이상
* 치명적 결함 0개
* 모든 Critical 기준이 M4 이상
* Critical 기준 중 `UNVERIFIED` 0개
* 공식 샘플 연구 종단 간 재현 성공
* 시점 정확성 적대 테스트 통과
* 생존편향 방지 테스트 통과
* 독립 재현 워크플로 확인
* 공식 산출물 불변성 확인
* 연구·실거래 경계 위반 없음

## NEAR COMPLETE

* 총점 90~94점
* 치명적 결함 0개
* Critical 기준 대부분 M4 이상
* 제한된 비핵심 공백만 존재

## SUBSTANTIALLY COMPLETE BUT INCOMPLETE

* 총점 80~89점
* 실질적 플랫폼 기능은 존재
* 하나 이상의 핵심 영역에 구조적 공백 존재

## FUNCTIONAL RESEARCH PLATFORM

* 총점 70~79점
* 연구 워크플로는 가능
* 데이터·검증·거버넌스·지식 관리 중 여러 요소 부족

## RESEARCH TOOLKIT

* 총점 50~69점
* 유용한 데이터·백테스트·분석 기능은 있음
* 통합 연구 생애주기 플랫폼은 아님

## PROTOTYPE

* 총점 30~49점
* 부분 구현 또는 데모 수준

## NOT AN INVESTMENT RESEARCH PLATFORM

* 총점 30점 미만
* 또는 실거래 중심 시스템
* 또는 치명적인 미래정보·재현성 문제가 존재

치명적 결함이 있으면 점수가 높더라도 `COMPLETE` 또는 `NEAR COMPLETE`로 판정하지 말라.
