첨부한 **레포 리뷰 평가기준 문서**를 이 작업의 최우선 기준이자 단일한 기준점으로 사용하여, 현재 레포를 진단하고 부족한 부분을 실제로 개선하라.

이 작업의 목표는 단순히 문제 목록을 작성하거나 일부 문제를 임시로 수정하는 것이 아니다.

최종 목표는 다음과 같다.

> 현재 레포가 첨부 문서의 모든 평가기준을 실질적으로 충족하도록, 진단·근본 원인 분석·구현·검증·재진단 과정을 반복하여 완성도를 끝까지 높인다.

## 1. 최우선 원칙

첨부한 평가기준 문서를 이 작업의 최우선 기준으로 삼아라.

레포의 기존 구조, 관례, 구현 방식, 문서, 테스트 또는 현재 코드가 평가기준 문서와 충돌한다면, 특별한 기술적 제약이 없는 한 평가기준 문서를 우선하라.

다만 다음과 같은 경우에는 임의로 충족했다고 판단하지 말고 구체적인 근거와 함께 명시하라.

- 평가기준끼리 서로 충돌하는 경우
    
- 현재 기술 스택에서 문자 그대로 구현하기 어려운 경우
    
- 외부 시스템, 비밀키, 인프라 또는 접근할 수 없는 자원이 필요한 경우
    
- 다른 기준을 심각하게 훼손하지 않고서는 충족할 수 없는 경우
    

이 경우에도 가능한 범위에서 가장 근접한 구조를 실제로 구현하고, 남은 제약과 필요한 후속 조치를 명확하게 기록하라.

## 2. 작업 권한과 범위

이 작업을 완료하기 위해 레포 내부의 모든 파일을 조사하고 변경할 수 있다.

필요하다면 다음 작업을 수행하라.

- 디렉터리와 모듈 구조 변경
    
- 파일 생성, 이동, 이름 변경, 통합 및 삭제
    
- 기존 구현의 대규모 리팩터링
    
- 인터페이스와 데이터 흐름 재설계
    
- 책임과 경계 재분리
    
- 의존성 구조 변경
    
- 설정 파일 수정
    
- 테스트 추가 및 수정
    
- 문서와 예제 수정
    
- 마이그레이션 코드 작성
    
- 사용되지 않는 코드와 중복 코드 제거
    
- 빌드, 테스트, 린트, 정적 분석 및 관련 검증 명령 실행
    

레포 내부 작업에 필요한 변경과 명령 실행은 승인된 것으로 간주하고, 사소한 변경마다 사용자에게 허가를 요청하지 말고 자율적으로 진행하라.

단, 레포 외부 시스템에 대한 배포, 실제 운영 데이터 변경, 비밀정보 사용, 외부 서비스의 파괴적 변경은 수행하지 말라.

## 3. 평가기준의 구조화

코드를 변경하기 전에 첨부 문서를 완전히 읽고, 문서의 내용을 실행 가능한 평가체계로 변환하라.

각 평가기준에 대해 최소한 다음을 정의하라.

- 기준의 정확한 의미
    
- 기준이 요구하는 이상적인 상태
    
- 코드에서 확인해야 할 대상
    
- 충족 여부를 판단할 수 있는 객관적 근거
    
- 완전 충족, 부분 충족, 미충족의 구분
    
- 다른 기준과의 의존관계
    
- 검증 방법
    
- 완료 조건
    

문서의 표현을 지나치게 축약하거나 자의적으로 완화하지 말라.

평가기준을 단순한 추상적 구호로 다루지 말고, 실제 코드와 구조를 판정할 수 있는 체크리스트와 평가 매트릭스로 변환하라.

## 4. 최초 진단

첫 번째 단계에서는 바로 코드를 수정하지 말고 레포 전체를 충분히 조사하라.

다음을 포함하여 레포의 현재 상태를 파악하라.

- 전체 디렉터리 구조
    
- 주요 진입점
    
- 핵심 모듈과 책임
    
- 계층과 경계
    
- 데이터 흐름
    
- 상태 관리 방식
    
- 외부 의존성
    
- 설정과 실행 방식
    
- 오류 처리
    
- 확장 지점
    
- 테스트 구조
    
- 문서와 실제 구현의 일치 여부
    
- 중복 책임과 숨겨진 결합
    
- 사용되지 않거나 우회된 코드
    
- 평가기준에 영향을 주는 모든 관련 파일
    

각 평가기준에 대해 현재 상태를 다음 중 하나로 판정하라.

- 완전 충족
    
- 부분 충족
    
- 미충족
    
- 판단 불가
    

모든 판정에는 반드시 관련 파일, 클래스, 함수, 모듈, 설정 또는 실행 결과와 같은 구체적인 근거를 연결하라.

## 5. 증상 수정이 아닌 근본 원인 해결

발견된 문제를 곧바로 개별적으로 수정하지 말라.

각 문제에 대해 다음 분석을 먼저 수행하라.

1. 표면에 드러난 문제는 무엇인가?
    
2. 그 문제가 발생한 직접적인 원인은 무엇인가?
    
3. 여러 문제가 공유하는 더 상위의 구조적 원인이 있는가?
    
4. 책임 배치, 경계, 추상화, 데이터 흐름, 의존성 방향 또는 설계 원칙 중 무엇이 잘못되었는가?
    
5. 현재 문제만 패치할 경우 어떤 유사 문제가 다시 발생할 수 있는가?
    
6. 근본 원인을 해결하려면 어느 수준까지 구조를 변경해야 하는가?
    
7. 해당 변경이 다른 평가기준에 미치는 영향은 무엇인가?
    

문제 하나마다 파일 하나를 수정하는 방식으로 접근하지 말라.

여러 문제의 공통 원인이 잘못된 구조, 책임 배치 또는 추상화에 있다면 개별 증상을 패치하지 말고 그 상위 구조를 재설계하라.

예를 들어 다음과 같은 수정은 피하라.

- 조건문 하나를 추가해 증상만 숨기는 수정
    
- 기존 구조를 유지한 채 래퍼만 덧붙이는 수정
    
- 평가를 통과하기 위한 형식적인 인터페이스 추가
    
- 실제로 사용되지 않는 추상화 추가
    
- 테스트만 통과시키기 위한 하드코딩
    
- 문서만 수정하고 실제 구현은 유지하는 방식
    
- 중복 구현을 남겨둔 채 새로운 구현을 병렬로 추가하는 방식
    
- 근본 원인을 남긴 채 예외 처리만 늘리는 방식
    

## 6. 개선 계획 수립

최초 진단과 근본 원인 분석이 끝난 후, 구현 전에 개선 계획을 세워라.

개선 계획에는 다음이 포함되어야 한다.

- 해결해야 할 근본 원인
    
- 영향을 받는 평가기준
    
- 변경할 구조와 책임
    
- 변경 대상 파일과 모듈
    
- 예상되는 파급 범위
    
- 기존 코드의 제거 또는 마이그레이션 방법
    
- 회귀 가능성
    
- 필요한 테스트와 검증 방법
    
- 작업 순서
    
- 각 작업의 완료 조건
    

작업 순서는 개별 문제의 수가 아니라 구조적 영향도와 선행관계를 기준으로 정하라.

가급적 다음 순서를 따른다.

1. 잘못된 경계와 책임 수정
    
2. 의존성 방향과 데이터 흐름 수정
    
3. 핵심 추상화 및 계약 정리
    
4. 세부 구현 수정
    
5. 오류 처리와 안정성 강화
    
6. 테스트와 검증 보강
    
7. 문서와 실제 구현 동기화
    
8. 불필요한 이전 구조 제거
    

## 7. 구현 원칙

계획이 수립되면 실제 코드를 끝까지 구현하라.

평가기준을 충족하는 것처럼 보이는 최소한의 변경이 아니라, 해당 기준의 목적을 실제로 달성하는 구현을 만들어라.

구현 과정에서는 다음 원칙을 지켜라.

- 임시방편보다 최종적으로 유지 가능한 구조를 선택한다.
    
- 부분 구현, 빈 함수, TODO, 주석으로 대체된 구현을 남기지 않는다.
    
- 사용되지 않는 인터페이스나 형식적인 추상화를 만들지 않는다.
    
- 새 구조를 추가했다면 기존 우회 경로와 중복 구조를 제거한다.
    
- 변경된 계약을 사용하는 모든 호출부를 함께 수정한다.
    
- 테스트가 실제 요구사항과 실패 조건을 검증하도록 한다.
    
- 정상 경로뿐 아니라 오류 경로와 경계 조건도 구현한다.
    
- 문서와 코드가 서로 다른 상태로 남지 않도록 한다.
    
- 평가기준 하나를 충족하면서 다른 기준을 훼손하지 않는다.
    
- 가능한 경우 정적 분석과 자동화된 테스트로 준수 여부를 고정한다.
    

## 8. 반복 진단과 개선

최초 진단을 1회차로 간주하고, 이후 다음 사이클을 최대 10회까지 반복하라.

### 반복 사이클

1. 첨부 문서의 전체 평가기준으로 현재 레포를 다시 진단한다.
    
2. 이전 회차에서 변경한 내용이 실제로 기준을 충족했는지 확인한다.
    
3. 새로 드러난 문제, 회귀, 불완전한 구현을 찾는다.
    
4. 표면 문제 뒤의 구조적 근본 원인을 다시 분석한다.
    
5. 근본 원인을 해결하는 방향으로 추가 구현한다.
    
6. 테스트, 빌드, 린트, 정적 분석과 필요한 실행 검증을 수행한다.
    
7. 전체 평가기준을 다시 평가한다.
    
8. 남아 있는 모든 부분 충족 및 미충족 항목을 다음 회차 작업에 반영한다.
    

반복 횟수를 억지로 10회까지 채우는 것이 목적은 아니다.

모든 평가기준이 객관적인 근거를 바탕으로 완전히 충족되었고, 추가 작업이 의미 있는 개선을 만들지 않는 상태라면 10회 이전에 종료할 수 있다.

반대로 아직 부분 충족이나 미충족 항목이 남아 있다면, 쉬운 문제만 해결하고 조기에 종료하지 말고 최대 10회까지 계속 개선하라.

매 회차는 단순히 같은 검사를 반복하는 것이 아니라, 이전 회차의 결과를 바탕으로 레포의 완성도를 실질적으로 높여야 한다.

## 9. 각 회차의 필수 기록

각 회차마다 다음 내용을 간결하지만 구체적으로 기록하라.

### 회차 진단

- 각 평가기준의 판정
    
- 판정 근거
    
- 남은 문제
    
- 새롭게 발견된 회귀 또는 구조적 문제
    

### 근본 원인 분석

- 표면 증상
    
- 직접 원인
    
- 상위 구조적 원인
    
- 선택한 해결 방향
    
- 단순 패치를 선택하지 않은 이유
    

### 구현 내용

- 변경한 구조
    
- 주요 변경 파일
    
- 제거한 이전 구조
    
- 변경된 데이터 흐름 또는 의존성
    
- 추가하거나 수정한 테스트
    

### 검증 결과

- 실행한 검증 명령
    
- 테스트 결과
    
- 빌드 결과
    
- 린트 및 정적 분석 결과
    
- 실패한 항목과 해결 여부
    

### 회차 종료 판정

- 완전 충족된 기준
    
- 아직 부분 충족인 기준
    
- 아직 미충족인 기준
    
- 다음 회차가 필요한 이유
    

## 10. 평가의 엄격성

평가기준을 느슨하게 해석하여 쉽게 통과시키지 말라.

다음과 같은 상태는 완전 충족으로 판정하지 말라.

- 일부 코드 경로에서만 기준을 만족하는 경우
    
- 새로운 구조가 추가되었지만 기존 우회 경로가 남아 있는 경우
    
- 테스트가 존재하지만 핵심 요구사항을 검증하지 않는 경우
    
- 인터페이스는 분리되었지만 실제 의존성이 그대로인 경우
    
- 파일은 분리되었지만 책임과 상태가 여전히 결합된 경우
    
- 문서상으로만 기준을 만족하는 경우
    
- 정상 경로만 구현되고 실패 경로가 방치된 경우
    
- 향후 구현을 전제로 TODO가 남은 경우
    
- 평가기준을 충족한다고 주장하지만 코드 근거가 없는 경우
    
- 테스트 실패, 빌드 실패 또는 주요 경고가 남아 있는 경우
    

평가를 통과하기 위한 코드를 만들지 말고, 평가기준이 요구하는 설계 목적과 품질 속성을 실제로 달성하라.

## 11. 회귀 방지

구조를 변경한 후에는 변경된 부분만 확인하지 말고 레포 전체를 다시 검증하라.

최소한 다음을 확인하라.

- 기존 기능이 유지되는가?
    
- 새로운 경계가 우회되지 않는가?
    
- 순환 의존성이 생기지 않았는가?
    
- 중복된 구현이 남아 있지 않은가?
    
- 사용되지 않는 코드가 남아 있지 않은가?
    
- 새 추상화가 실제로 사용되는가?
    
- 테스트가 구현 세부사항이 아닌 요구사항을 검증하는가?
    
- 설정, 문서, 예제와 실제 코드가 일치하는가?
    
- 한 평가기준의 개선이 다른 평가기준을 악화시키지 않았는가?
    

가능하다면 평가기준 준수를 자동으로 검증할 수 있는 테스트, 아키텍처 테스트, 정적 규칙 또는 검증 스크립트를 추가하라.

## 12. 종료 조건

다음 조건이 모두 충족될 때만 작업을 완료된 것으로 판단하라.

- 첨부 문서의 모든 평가기준을 다시 검토했다.
    
- 모든 기준이 완전 충족으로 판정되었다.
    
- 모든 판정에 코드 또는 실행 결과에 기반한 근거가 있다.
    
- 부분 충족 및 미충족 항목이 남아 있지 않다.
    
- 구조적 근본 원인이 해결되었다.
    
- 임시방편, 미완성 코드, TODO 및 사용되지 않는 대체 구현이 남아 있지 않다.
    
- 전체 테스트가 통과한다.
    
- 빌드가 성공한다.
    
- 린트와 정적 분석에서 작업과 관련된 해결되지 않은 문제가 없다.
    
- 주요 오류 경로와 경계 조건이 검증되었다.
    
- 코드, 테스트, 설정, 문서가 서로 일치한다.
    
- 최종 재진단에서 새로운 회귀가 발견되지 않았다.
    

최대 10회의 반복 이후에도 충족하지 못한 기준이 있다면 이를 숨기거나 완전 충족으로 과장하지 말라.

남은 항목마다 다음을 명확히 작성하라.

- 충족하지 못한 평가기준
    
- 현재 충족 수준
    
- 남은 구체적인 문제
    
- 해결하지 못한 이유
    
- 필요한 외부 조건 또는 추가 작업
    
- 현재 코드에서 확인할 수 있는 근거
    

## 13. 최종 보고서

작업이 끝나면 다음 형식으로 최종 결과를 보고하라.

### 1. 최종 결론

- 전체 평가 결과
    
- 수행한 반복 횟수
    
- 최종적으로 완전 충족한 기준 수
    
- 남아 있는 부분 충족 또는 미충족 기준 수
    

### 2. 평가기준별 최종 상태

각 기준에 대해 다음을 작성한다.

- 최종 판정
    
- 구현된 내용
    
- 관련 파일과 핵심 코드
    
- 검증 근거
    
- 관련 테스트
    

### 3. 해결한 근본 원인

- 최초에 발견된 주요 증상
    
- 실제 상위 구조적 원인
    
- 적용한 구조적 해결책
    
- 단순 패치와 비교했을 때 개선된 점
    

### 4. 주요 변경 사항

- 구조 변경
    
- 책임과 경계 변경
    
- 데이터 흐름 변경
    
- 의존성 변경
    
- 제거한 레거시 구조
    
- 추가한 테스트와 검증 장치
    
- 수정한 문서
    

### 5. 검증 결과

- 실행한 명령
    
- 테스트 결과
    
- 빌드 결과
    
- 린트 결과
    
- 정적 분석 결과
    
- 추가 검증 결과
    

### 6. 남은 문제

남은 문제가 없다면 명시적으로 “없음”이라고 작성한다.

문제가 있다면 심각도, 관련 평가기준, 근거, 원인 및 필요한 후속 작업을 작성한다.

## 14. 즉시 실행

이 프롬프트에 대한 계획만 설명하거나 사용자에게 작업 진행 여부를 다시 묻지 말라.

지금부터 다음 순서로 작업을 시작하라.

1. 첨부 평가기준 문서 전체 분석
    
2. 실행 가능한 평가 매트릭스 작성
    
3. 레포 전체 조사
    
4. 최초 진단
    
5. 문제별 근본 원인 분석
    
6. 구조적 개선 계획 수립
    
7. 실제 구현
    
8. 전체 검증
    
9. 평가기준에 따른 재진단
    
10. 필요한 경우 최대 10회까지 반복
    
11. 최종 보고서 작성
    

가장 중요한 것은 반복 횟수나 수정한 파일 수가 아니다.

**첨부 문서의 평가기준을 정확하게 이해하고, 문제의 표면이 아니라 상위 구조적 근본 원인을 해결하며, 레포가 모든 기준을 실질적으로 충족할 때까지 구현과 검증을 완결하는 것​**이 이 작업의 핵심이다.
