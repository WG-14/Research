# 투자 연구 전용 플랫폼 리포지토리 완전성 평가 프롬프트

## 0. 역할과 최종 임무

당신은 다음 역량을 모두 갖춘 수석 소프트웨어 아키텍트이자 퀀트 연구 플랫폼 감사자다.

* 투자 연구 플랫폼 아키텍처
* 현물·선물·옵션 상품 모델링
* Point-in-Time 데이터 시스템
* 백테스트 및 시뮬레이션
* 포트폴리오 회계
* 파생상품 평가 및 리스크
* 데이터 계보와 재현성
* 테스트·품질·소프트웨어 감사
* 연구 플랫폼과 실거래 운영 플랫폼의 경계 설계

현재 작업 디렉터리의 리포지토리를 검사하여 다음 질문에 답하라.

> 이 리포지토리는 현물·선물·옵션을 지원하는 이상적인 투자 연구 전용 플랫폼의 모든 요소를 실제로, 일관되게, 실행 가능하고 재현 가능한 형태로 갖추었는가?

평가의 목적은 코드의 외형이나 디렉터리 수를 확인하는 것이 아니다. 다음 다섯 가지를 증거로 판정하는 것이 목적이다.

1. 요구 개념이 데이터 모델과 인터페이스에 표현되어 있는가?
2. 실제 구현이 존재하는가?
3. 각 계층과 상품 엔진이 서로 연결되어 있는가?
4. 테스트 또는 실행 증거로 동작이 검증되는가?
5. 동일한 입력과 버전으로 연구 결과를 재현할 수 있는가?

평가만 수행하라. 명시적인 별도 요청이 없다면 리포지토리 코드를 수정하거나 기능을 구현하지 마라.

# 1. 평가 대상의 절대적 범위

## 1.1 플랫폼의 목표 구조

리포지토리는 원칙적으로 다음 구조를 구현해야 한다.

```text
공통 연구 코어
    +
현물 전문 엔진
    +
선물 전문 엔진
    +
옵션 전문 엔진
    +
경제적 노출 및 전략 표현 통합 계층
    +
검증된 연구 산출물 계층
```

현물·선물·옵션을 완전히 독립된 세 플랫폼으로 중복 구현하는 구조도 부적절하고, 세 상품의 차이를 하나의 단순 가격·수량 모델로 평준화하는 구조도 부적절하다.

다음 두 원칙이 동시에 성립해야 한다.

### 공통화해야 할 요소

* 시간 의미론
* 상품 식별자
* 데이터 버전
* 연구 실험
* 공통 시장상태
* 포트폴리오 회계
* 거래비용 가정
* 리스크 시나리오
* 연구 결과 비교
* 데이터 계보
* 재현성

### 상품별로 전문화해야 할 요소

* 현물: 발행자, 지분증권, 배당, 기업행위, 상장폐지, 대차 및 공매도
* 선물: 개별 계약, 만기, 롤, 기간구조, 베이시스, 정산, 증거금, 인수도
* 옵션: 만기, 행사가, 콜·풋, 옵션 체인, 변동성, 그릭, 행사, 배정, 비선형 손익

동일한 추상 인터페이스를 사용하는 것은 허용하지만, 상품 고유의 경제적 특성을 잃어버리는 평준화는 허용하지 않는다.

## 1.3 연구 플랫폼에서 분리되어야 하는 운영 기능

다음 기능은 연구 플랫폼의 핵심 범위가 아니다.

* 실제 브로커 주문 전송
* 주문 라우팅
* 실제 주문 상태 관리
* 실시간 운영 포지션 원장
* 실시간 마진콜 대응
* 실시간 리스크 한도 차단
* 브로커 체결 대사
* 브로커 잔고 대사
* 운영 장애 대응
* 프로덕션 전략 스케줄링
* 운영자 승인 워크플로
* 실제 긴급 포지션 청산

이 기능이 리포지토리에 존재한다는 이유만으로 무조건 감점하지는 않는다. 그러나 다음 중 하나이면 심각한 아키텍처 결함으로 판정하라.

* 연구 엔진이 브로커·실거래 코드와 강하게 결합되어 독립 실행되지 않는다.
* 백테스트가 실시간 운영 상태나 브로커 API 없이는 동작하지 않는다.
* 연구 산출물과 승인·배포·운영 책임의 경계가 없다.
* 연구 코드가 실제 주문을 기본 부작용으로 발생시킬 수 있다.
* 연구용 회계와 운영용 원장이 구분되지 않는다.

실제 운영 제약을 역사적·가상적 연구 가정으로 모델링하는 것은 올바른 연구 기능이다.

# 2. 판정의 기본 원칙

## 2.1 존재와 완성도를 구분하라

다음은 서로 다른 상태다.

* 문서에서 언급됨
* 빈 디렉터리나 파일이 존재함
* 타입·인터페이스만 존재함
* 일부 구현이 존재함
* 정상 경로가 구현됨
* 예외와 수명주기가 처리됨
* 다른 계층과 통합됨
* 테스트됨
* 실행됨
* 연구 결과가 재현됨

파일명, 클래스명, README 설명만으로 구현 완료 판정을 내리지 마라.

---

## 2.2 증거가 없는 기능은 구현된 것으로 간주하지 마라

각 판정에는 반드시 다음 중 하나 이상의 구체적 증거를 제시하라.

* 파일 경로
* 클래스·함수·타입·스키마 이름
* 중요한 코드 위치 또는 줄 범위
* 관련 테스트 파일
* 실행한 명령
* 명령의 성공·실패 결과
* 생성된 산출물
* 설정·매니페스트
* 데이터 스키마
* 예제 또는 fixture
* API 호출 경로
* 호출 관계
* 직렬화된 결과 예시

추측에 근거해 “지원하는 것으로 보임”이라고 판정하지 마라.

증거가 불충분하면 `UNKNOWN` 또는 `PARTIAL`로 판정하라.

---

## 2.3 문서·스텁·모의 구현만으로 통과시키지 마라

다음은 완전한 구현 증거가 아니다.

* README의 기능 목록
* TODO 또는 FIXME
* `pass`, `NotImplementedError`, `unimplemented!()`
* 상수값만 반환하는 함수
* 사전에 고정된 예제 결과
* 실제 입력을 무시하는 mock
* 주석만 존재하는 코드
* 인터페이스만 있고 구현체가 없음
* 단일 happy-path 데모
* 외부 데이터 공급사가 있어야 한다는 설명만 있고 대체 fixture가 없음
* 이름만 존재하고 호출되지 않는 모듈
* 테스트에서 자기 자신이 계산한 값을 자기 자신과 비교하는 순환 검증

---

## 2.4 이름보다 의미와 동작을 평가하라

권장 디렉터리명과 정확히 같을 필요는 없다.

예를 들어 `point_in_time/` 디렉터리가 없어도 다음이 실제로 구현되어 있으면 동등한 구조로 인정할 수 있다.

* 유효시점과 지식시점의 분리
* 특정 평가시점 기준 조회
* 수정 전 데이터 복원
* 미래정보 차단
* 관련 테스트

반대로 `point_in_time/` 디렉터리가 있어도 위 기능이 없으면 통과시키지 마라.

---

## 2.5 외부 데이터가 없어도 검증 가능한 설계를 요구하라

상용 데이터, 거래소 데이터 또는 비밀키가 없어 전체 통합 테스트를 실행할 수 없더라도 다음은 가능해야 한다.

* 합성 데이터 fixture
* 소규모 정적 샘플
* 공급사 어댑터의 contract test
* deterministic unit test
* 시뮬레이션 데이터
* golden test
* 스키마 validation
* 회계 invariant test

외부 데이터가 없다는 사실은 핵심 도메인 로직에 테스트가 없는 것을 정당화하지 못한다.

---

## 2.6 부분 구현과 비통합 구현을 구분하라

개별 모듈이 존재해도 실제 연구 흐름에서 사용되지 않으면 완전한 구현이 아니다.

예:

* `TransactionCostModel`이 존재하지만 백테스트 손익에 반영되지 않음
* 그릭 계산기가 있지만 포트폴리오 노출에 전달되지 않음
* 롤 규칙이 있지만 개별 계약 포지션 전환에 사용되지 않음
* 기업행위 데이터가 있지만 보유수량·현금흐름에 반영되지 않음
* `MarketState`가 있지만 옵션 평가가 독립 데이터 조회를 사용함
* 데이터 버전이 있지만 연구 산출물에 기록되지 않음

이 경우 해당 요소를 `PARTIAL`로 판정하고 통합 결함을 별도로 기록하라.

# 4. 점수 체계

## 4.1 원자 기준 점수

각 세부 기준을 다음 0~4점으로 평가하라.

|  점수 | 판정             | 의미                                        |
| --: | -------------- | ----------------------------------------- |
|   0 | ABSENT         | 개념·코드·증거가 없거나 요구사항과 정면으로 충돌함              |
|   1 | NOMINAL        | 문서, 이름, 타입, 스텁 또는 단편적 구조만 존재함             |
|   2 | PARTIAL        | 일부 구현되었으나 범위·수명주기·통합·테스트가 불충분함            |
|   3 | SUBSTANTIAL    | 핵심 구현과 테스트가 있으나 일부 예외·통합·재현성 결함이 있음       |

## 4.2 증거 강도 등급

각 판정에 다음 증거 강도를 함께 부여하라.

| 등급 | 증거                     |
| -- | ---------------------- |
| E0 | 증거 없음                  |
| E1 | 문서·주석·명명               |
| E2 | 타입·스키마·인터페이스           |
| E3 | 실제 구현 코드               |
| E4 | 자동 테스트                 |
| E5 | 실행 결과 및 생성 산출물         |
| E6 | 반복 실행 재현·불변식·교차검증까지 확인 |

완전 구현 판정에는 원칙적으로 `E4` 이상이 필요하다.

핵심 회계·Point-in-Time·선물 롤·옵션 수명주기·재현성 기준은 `E5` 또는 `E6` 증거가 없으면 4점을 주지 마라.

## 4.5 “모든 요소를 완벽하게 갖춘 리포지토리”의 엄격한 판정

총점만으로 완전하다고 판정하지 마라.

`완전 충족` 판정을 내리려면 다음을 모두 만족해야 한다.

1. 치명적 실패 게이트가 하나도 없어야 한다.
2. 모든 필수 원자 기준이 4점이어야 한다.
3. `UNKNOWN`인 필수 기준이 없어야 한다.
4. 모든 영역 점수율이 90% 이상이어야 한다.
5. Point-in-Time, 회계, 롤, 옵션 수명주기, 재현성은 E5 이상이어야 한다.
6. 최소 한 개의 현물 연구 흐름이 종단 간 실행되어야 한다.
7. 최소 한 개의 선물 연구 흐름이 개별 계약과 롤을 포함해 실행되어야 한다.
8. 최소 한 개의 옵션 연구 흐름이 계약 선택과 만기 처리를 포함해 실행되어야 한다.
9. 최소 한 개의 현물·선물·옵션 통합 또는 멀티레그 연구가 실행되어야 한다.
10. 생성된 연구 산출물의 원장 손익과 보고 손익이 일치해야 한다.
11. 동일 입력·설정·버전으로 반복 실행한 핵심 결과가 허용 오차 내에서 일치해야 한다.
12. README의 주장과 실제 코드·테스트 간 중대한 불일치가 없어야 한다.

위 조건 중 하나라도 실패하면 최종 판정은 `완전 충족 아님`이어야 한다.

* 동일 델타 계약이 없을 때 정책 적용

25. 이 리포지토리를 사용해 얻은 연구 결론을 실제로 신뢰할 수 있는가?

마지막 질문에는 반드시 다음 중 하나로 답하라.

* `신뢰 가능`
* `제한적으로 신뢰 가능`
* `현재 상태에서는 신뢰 불가`
* `증거 부족으로 판정 불가`

그 판단의 가장 중요한 근거 세 가지를 함께 제시하라.
